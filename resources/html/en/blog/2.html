The programming languages that I chose

Before starting the Systems Analysis and Development course, I already had used some programming languages for my personal projects, like Pascal, Python, C and JavaScript, specially these two last languages. But, once I started the course, I aimed to study the most demanded languages in the job market, the ones widely used for the back-end.

My first task was to get more professional in JavaScript and Node.js, because I already knew the language and I wanted to fill the gaps of my knowledge in regard to it. Months later, when I started studying front-end frameworks, more precisely Angular, I also studied TypeScript.

The second and most important language was Java with Spring Boot, but in the course of the learning process, I studied JDBC, Hibernate and the core of Spring Framework before studying Spring Boot. I said the “most important” language because my plan was that Java would be my main language, the one that would really put food on my table.

As an aspiring back-end developer, it's logic that I studied relational and non-relational databases, like MySQL/MariaDB, PostgreSQL and MongoDB.

Because of the faculty, I ended (re)studying other languages like Python. But none of them actually become an official part of my stack.

For a time, I wanted to create native mobile phone applications, because of that I studied Kotlin, but my computer could not execute Android Studio in a comfortable way, and soon later I noticed that I actually didn't have many ideas for mobile phone apps to create. I also saw a bit of C#, because it was the most used back-end language after Java. Studying it along .NET would increase my chances of getting my first job in the industry. Oh, I almost forgot, some months ago I studied C++ too, a language that I used in my teenage years, the idea was to compile the C++ code to WebAssembly through Emscripten.

My dilemma was the language that I would use in my personal projects; beyond JavaScript, of course. Before we continue, let me explain why not Java: it's because Java is terribly heavyweight for the applications I wanted to make, except for Android apps and non-trivial server side apps, not to mention that the user would need to install the JRE. At the end, I decided that my “toy language” should run on the browser, no matter if compiled into WebAssembly to run in the client side, or executed in the server side. The two candidate languages were Rust and Go, I listed just below the reasons to choose each one, and just after their respective counter-arguments:

RUST

  • Ideal for the creation of applications that run in WebAssembly. And I like to create programs.
    • On the other hand, the majority of my applications can be made in JavaScript, and if I need to create something complex, maybe the best is that I run it in the server side.
    • I have many ideas of applications, but do I really want to create them? Do I really want to create an offline text RPG, or a typesetting tool like SILE or LaTeX, or would it be the best to dedicate my free time to write my books? And I still can make these software in Go...
  • Despite the fact that there aren't many job opportunities in this language, its community is very active and engaged, I can imagine Rust being maintained in the next decades, unless the paradigm of everything changes, like the case in which the artificial intelligence takes charge of everything. In other words, the language has a good prospect of longevity.
    • As I said, there are few job opportunities. In the time I asked myself which language to choose, Go offered at least six times more job openings.
    • For a time, I had this idea of creating a project that would last decades, like a text RPG that I would maintain my whole life, but I started changing my vision, like in focusing more in my artistic projects, and creating programming projects simple enough that I could rewrite them in another language if it were necessary.

GOLANG

  • I have an interest in DevOps, and Go is the language that has space in this area.
    • However, other languages tend to be more utilized, like Java/C# or Python
  • There are more job openings for Go than for Rust.
    • But, if it is for me getting a programmer job, much probably it will be thanks Java, JavaScript or even C#.
      • But Go is still a language that can give me a job...
  • And even in GitHub, there are much more Go projects, a good example is that I couldn't find a PDF library in Rust!
  • All dependencies stay inside a single executable.
    • Maybe this is not the ideal for creating WebAssembly applications.
      • A counter-argument for this counter-argument is that probably I would use Go in the server side anyway.
      • If the project needs to be made in WebAssembly, it's because it is a complex program, in this case the user can wait a few more seconds for the program to load
      • TinyGo...
  • Important projects like Docker, Kubernetes, Terraform, CockroachDB, and Hugo were written in Go. I should believe in the people that said that Go is here to stay.
  • Well, I intend to be a back-end developer, it's reasonable that I should focus on languages made for running predominately in the server side 🙂

I know that the comparison above seems strange, it was like it were a discussion, but it really is that. From time to time I focused on Rust, and in another one I focused on Go. Writing (and re-reading) this comparison helped me in my choice. Why not learn both languages? Because I didn't want to study many languages at the same time, and I needed to focus on those that would give me a job!

Looking at the amount of job openings, I could choose C++ instead Rust/Go, but these two languages are modern and easy to work with other libraries and frameworks, not to mention that C++ is more used with embedded systems, not so much with web development where there is the majority of the job openings...

After a lot of thinking, I reached to the conclusion that Java would be the ideal for projects that need to run the whole time, like a MUD, and Go for services that only need to run from time, like a software that manipulates PDFs. I have no technical justification for this, this is just an excuse for using Java in my personal projects too. So I decided myself for this list of languages:

  • Java: It was always my professional focus
  • JavaScript: High demand in the job market, and widely used in my personal projects
  • Go: For my personal projects when Java and JavaScript were not convenient
  • A bit of C#, but only to be able to compete for junior or internship openings with this language
