The programming languages that I chose

Before starting the Systems Analysis and Development course, I already had used some programming languages for my personal projets, like Pascal, Python, C and JavaSript, specially these two last languages. But, as I started the course, I aimed to study the most demanded languages in the job market, the ones widely used in the back-end.

I started getting a bit more professional in JavaScript and Node.js, because I already knew the language and I wanted to fill the gaps of my knowledge in regard to it. Months later, when I started studying front-end frameworks, more precisely Angular, I also studied TypeScript.

The second and most important language was Java and Spring Boot, but in the course of the learning, I studied JDBC, Hibernate and the core of Spring Framework before studying Spring Boot. I said the “most important” language because my plan was that Java would be my main language, the one that would really put food on my table.

As a aspiring back-end developer, it's logic that I studyed relational and non-relational databases, like MySQL/MariaDB, PostgreSQL and MongoDB.

Because of the faculty, I ended (re)studying other languages like Python. But none of them actually become official part of my stack.

For a time, I wanted to create native mobile phone applications, because of that I studyed Kotlin, but my computer did not executed Android Studio in a comfortable way, and soon later I noticed that I actually didn't have many ideas for mobile phone apps to create. I also saw a bit of C#, because it was the most used back-end language after Java. Study it along the .NET would increase my chances of getting my first job in the industry. Oh, I almost forgot, some months ago I studyed C++ too, a language that I used in my teenage years, the idea was to compile the C++ code to WebAssembly through Emscripten.

My dilema was with the language that I would use in my personal projects; beyond JavaScript, of course. Before we continue, let me explain why not Java: it's because Java is terribly heavyweight for the applications I wanted to make, except for Android apps, not to mention that the user would need to install the JRE. At the end, I decided that my “toy language” should run on the browser, no matter if compiled into WebAssembly to run in the client side, or executed in the server side. The two candidate languages were Rust and Go, I listed just below the reasons to choose each one and just after their respective counter-argument:

RUST

  • Ideal para criação de aplicações que rodam em WebAssembly em todos os sentidos. E como gosto de criar programas, seria o ideal.
    • Por outro lado, a maioria das minhas aplicações podem ser feitas em JavaScript e se eu precisar criar algo complexo, talvez seja melhor que rode no lado do servidor.
    • Tenho muitas ideias de aplicações, mas eu realmente quero mesmo criá-las? Realmente eu quero mesmo criar um RPG de texto offline ou ferramenta de typesetting como SILE ou LaTeX, ou seria melhor eu dedicar o meu tempo livre mais para escrever meus livros?
  • Apesar de não haver muitas oportunidades de emprego nessa linguagem, a sua comunidade era muito ativa e engajada, eu conseguia imaginar o Rust sendo mantido nas próximas décadas, a não ser que o paradigma de tudo mudasse, como a inteligência artificial tomando conta de tudo. Ou seja, a linguagem tinha um bom prospecto de longetividade.
    • Como eu disse, havia poucas oportunidades de emprego. Na época em que eu me perguntava qual linguagem escolher, Go oferecia, no mínimo, duas vezes mais vagas.
    • Na época, eu tinha muito a ideia de criar projeto que eu manteria por décadas, como um RPG de texto que eu manteria por toda a vida, mas comecei a mudar de opinião, como me forcar mais nos meus projetos artísticos e criar projetos de programação que eu poderia reescrever em outra linguagem se a necessidade exigisse.

GOLANG

  • Eu tinha interesse em DevOps e Go era uma linguagem que tinha espaço nessa área.
    • Entretanto, outras linguagens tendiam a ser mais utilizadas, como Java/C# ou Python
  • Na época, havia mais vagas para Go do que para Rust
    • Mas, se fosse para eu conseguir um emprego como programador, muito provavelmente seria graças ao Java, JavaScript ou até mesmo C#.
      • Mas o Go não deixava de ser uma linguagem que poderia me dar um emprego...
  • Todas as dependências ficavam em um único executável.
    • Talvez não seja o ideal para criar aplicações em WebAssembly.
      • Um contra-argumento desse contra-argumento é que provavelmente eu usaria o Go no lado do servidor mesmo.
      • TinyGo...
  • Projetos importantes como Docker, Kubernetes, Terraform, CockroachDB, Hugo foram escritos em Go. Eu deveria acreditar no que o pessoal disse que o Go estava aqui para ficar
  • Bom, eu pretendia ser um desenvolvedor back-end, é normal que eu deveria me focar em linguagens feitas para rodar mais predominantemente no lado do servidor 🙂
