As linguagens de programação que escolhi

Antes de iniciar o curso de Análise e Desenvolvimento de Sistemas, eu já havia usado algumas linguagens de programação nos meus projetos pessoais, como Pascal, Python, C e JavaScript, especialmente essas duas últimas linguagens. Mas, assim que iniciei o curso, procurei estudar as linguagens com mais demanda no mercado e que fossem muito usadas para o back-end.

Minha primeira tarefa foi me profissionalizar um pouco mais no JavaScript e Node.js, pois eu já conhecia a linguagem e queria cobrir as lacunas do meu conhecimento a respeito dela. Meses mais tarde, quando comecei a estudar frameworks de front-end, mais precisamente o Angular, também estudei TypeScript.

A segunda e mais importante linguagem foi Java com Spring Boot, mas no decorrer do aprendizado, estudei JDBC, Hibernate e o core do Spring Framework antes de estudar o Spring Boot. Eu disse linguagem “mais importante” porque meu planejamento foi de que seria Java a minha principal linguagem, a que realmente colocaria comida na minha mesa.

Como aspirante a desenvolvedor back-end, é lógico que estudei bancos de dados relacionais e não-relacionais, como MySQL/MariaDB, PostgreSQL e MongoDB.

Por causa da faculdade, acabei (re-)estudando outras linguagens como Python. Mas nenhuma delas de fato se tornaram parte oficial da minha stack.

Por um tempo, tive vontade de criar aplicativos para celular nativos, por conta disso estudei Kotlin, mas meu computador não rodava o Android Studio de forma confortável e logo depois percebi que eu, na verdade, não tinha muitas ideias de aplicativos de celular para criar. Também vi um pouco de C#, pois era a linguagem mais usada no back-end depois do Java. Estudá-la junto do framework .NET aumentaria minhas chances de conseguir meu primeiro emprego na indústria. Oh, quase me esqueci, meses antes eu também tinha estudado C++, uma linguagem que mexi na adolescência, a ideia principal era compilar o código em C++ para WebAssembly através do Emscripten.

Meu dilema foi com a linguagem que eu usaria nos meus projetos pessoais; além do JavaScript, é claro. Antes de continuarmos, já explico porque não seria o Java: é porque Java é terrivelmente pesado para as aplicações que eu queria fazer, exceto para aplicativos para Android e aplicações não triviais que rodam no lado do servidor, sem contar que o usuário precisaria baixar a o JRE. No fim, decidi que minha “toy language” rodaria no navegador, seja compilada em WebAssembly para rodar no lado do cliente ou sendo executada no lado do servidor. As duas linguagens candidatas eram Rust e Go, listei abaixo os motivos para escolher cada uma delas e logo a seguir seus respectivos contra-argumentos:

RUST

  • Ideal para criação de aplicações que rodam em WebAssembly. E como gosto de criar programas...
    • Por outro lado, a maioria das minhas aplicações podem ser feitas em JavaScript e se eu precisar criar algo complexo, talvez seja melhor que rode no lado do servidor.
    • Tenho muitas ideias de aplicações, mas eu realmente quero mesmo criá-las? Realmente eu quero mesmo criar um RPG de texto offline ou ferramenta de typesetting como SILE ou LaTeX, ou seria melhor eu dedicar o meu tempo livre mais para escrever meus livros?
    • E eu ainda posso fazer esses softwares em Go...
  • Apesar de não haver muitas oportunidades de emprego nessa linguagem, a sua comunidade é muito ativa e engajada, eu consego imaginar o Rust sendo mantido nas próximas décadas, a não ser que o paradigma de tudo mude, como a inteligência artificial tomando conta de tudo. Ou seja, a linguagem tem um bom prospecto de longetividade.
    • Como eu disse, havia poucas oportunidades de emprego. Na época em que eu me perguntava qual linguagem escolher, Go oferecia, no mínimo, seis vezes mais vagas.
    • Por um tempo, eu tinha muito a ideia de criar projeto que eu manteria por décadas, como um RPG de texto que eu manteria por toda a vida, mas comecei a mudar de visão, como me forcar mais nos meus projetos artísticos e criar projetos de programação simples o suficiente para que eu pudesse reescrevê-los em outra linguagem se a necessidade exigisse.

GOLANG

  • Eu tenho interesse em DevOps e Go é uma linguagem que tem espaço nessa área.
    • Entretanto, outras linguagens tendem a ser mais utilizadas, como Java/C# ou Python
  • Há mais vagas para Go do que para Rust.
    • Mas, se é para eu conseguir um emprego como programador, muito provavelmente será graças ao Java, JavaScript ou até mesmo C#.
      • Mas o Go não deixa de ser uma linguagem que pode me dar um emprego...
  • E até no GitHub, há muito mais projetos em Go, um bom exemplo é que não pude encontrar uma biblioteca para PDF em Rust!
  • Todas as dependências ficam em um único executável.
    • Talvez não seja o ideal para criar aplicações em WebAssembly.
      • Um contra-argumento desse contra-argumento é que provavelmente eu usaria o Go no lado do servidor mesmo.
      • Se o projeto precisa ser feito em WebAssembly, é porque ele é um programa complexo, nesse caso o usuário pode esperar uns poucos segundos a mais para o programa carregar
      • TinyGo...
  • Projetos importantes como Docker, Kubernetes, Terraform, CockroachDB e Hugo foram escritos em Go. Eu deveria acreditar no que o pessoal disse que o Go estava aqui para ficar
  • Bom, eu pretendo ser um desenvolvedor back-end, é razoável que eu deveria me focar em linguagens feitas para rodar mais predominantemente no lado do servidor 🙂

Sei que a comparação acima pareceu estranha, como se fosse uma discussão, mas ela é realmente isso. De tempos em tempos eu me focava no Rust e em outros no Go. Escrever (e reler) esta comparação me ajudou na escolha. Por que não aprender ambas as linguagens? Porque eu não queria estudar muitas ao mesmo tempo e eu precisava me focar naquelas que me dariam um emprego!

Olhando a quantidade de vagas, eu poderia escolher o C++ em vez do Rust/Go, mas essas duas linguagens são modernas e mais fáceis de trabalhar com outras bibliotecas e frameworks, sem contar que o C++ é mais utilizado com embarcados, não tanto com desenvolvimento web onde ficam a maioria das vagas...

Após muito pensar, cheguei a conclusão de que Java seria o ideal para projetos que precisam rodar o tempo todo, como um MUD, e Go para serviços que só precisam funcionar de tempos em tempos, como um software que manipula PDFs. Não tenho uma justificativa técnica pra isso, apenas uma desculpa para usar Java também em meus projetos pessoais. Sem contar que eu pretendia usar o Go para criar programas em WebAssembly. Então decidi-me por esta lista de linguagens:

Em resumo:

  • Java: Nunca deixou de ser meu foco profissional
  • JavaScript: Alta demanda de mercado e muito usada nos meus projetos pessoais
  • Go: Para projetos em WebAssembly e back-end
  • Tecnicamente eu deveria adicionar C a esta lista, mas não uso mais esta linguagem ativamente...
